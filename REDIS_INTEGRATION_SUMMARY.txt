================================================================================
REDIS CACHING IN PHASE 1
================================================================================

What Gets Cached:

1. Active Graph Versions
   - Key: graph:active:{building_id}
   - TTL: 1 hour (3600 seconds)
   - Performance: ~500ms → ~10ms (50x faster)

2. Building Data
   - Key: building:{building_id}
   - TTL: 24 hours (86400 seconds)
   - Performance: ~200ms → ~10ms (20x faster)

3. Floor Data
   - Key: floor:{floor_id}
   - TTL: 24 hours (86400 seconds)
   - Performance: ~200ms → ~10ms (20x faster)

PERFORMANCE IMPACT
================================================================================

Without Caching:
- Route calculation: ~500ms

ENVIRONMENT CONFIGURATION
================================================================================

Add to .env:

# Redis Configuration
REDIS_URL=redis://localhost:6379
REDIS_DB=0
CACHE_ENABLED=true

# Cache TTL Values (in seconds)
CACHE_TTL_GRAPH=3600          # 1 hour for graph versions
CACHE_TTL_BUILDING=86400      # 24 hours for building data
CACHE_TTL_FLOOR=86400         # 24 hours for floor data

================================================================================
CACHE INVALIDATION STRATEGY
================================================================================

When to Invalidate:

1. Graph Cache
   - When graph is rebuilt: POST /api/graphs/rebuild/{building_id}
   - When graph is confirmed: POST /api/graphs/confirm/{building_id}
   - When graph is rolled back: POST /api/graphs/rollback/{building_id}

2. Building Cache
   - When building is created: POST /bff/buildings
   - When building is updated: PUT /bff/buildings/{id}

3. Floor Cache
   - When floor is created: POST /bff/floors
   - When floor is updated: PUT /bff/floors/{id}

KEY POINTS
================================================================================

1. Cache is optional but recommended
   - Improves performance significantly (4-5x faster)
   - Reduces load on external services (80% fewer calls)

2. Cache invalidation is critical
   - Keep cache fresh by invalidating on updates
   - Use consistent cache key format

3. Use async Redis client
   - aioredis for non-blocking I/O
   - Consistent with FastAPI async design

4. Monitor cache performance
   - Track cache hit rate (target: > 80%)
   - Monitor response times
   - Log cache operations

5. Test cache behavior
   - Unit tests for cache client
   - Integration tests for cache with services
   - Test cache hit/miss scenarios

6. Handle cache failures gracefully
   - Fall back to direct calls if cache unavailable
   - Log cache errors
   - Don't let cache failures break the app

7. Document cache keys
   - Use consistent format: {entity}:{id}
   - Document TTL values
   - Document invalidation strategy

8. Log cache operations
   - Log cache hits and misses
   - Log cache invalidation
   - Log cache errors
   - Use structured logging
