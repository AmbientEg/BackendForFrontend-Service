================================================================================
REDIS CACHING OVERVIEW
================================================================================

What Gets Cached:

1. Active Graph Versions
   - Key: graph:active:{building_id}
   - TTL: 1 hour (3600 seconds)
   - Performance: ~500ms → ~10ms (50x faster)
   - Use case: Avoid repeated calls to get active graph

2. Building Data
   - Key: building:{building_id}
   - TTL: 24 hours (86400 seconds)
   - Performance: ~200ms → ~10ms (20x faster)
   - Use case: Cache building metadata

3. Floor Data
   - Key: floor:{floor_id}
   - TTL: 24 hours (86400 seconds)
   - Performance: ~200ms → ~10ms (20x faster)
   - Use case: Cache floor metadata

================================================================================
CACHE INVALIDATION STRATEGY
================================================================================

When to Invalidate:

1. Graph Cache
   - When graph is rebuilt: POST /api/graphs/rebuild/{building_id}
   - When graph is confirmed: POST /api/graphs/confirm/{building_id}
   - When graph is rolled back: POST /api/graphs/rollback/{building_id}

2. Building Cache
   - When building is created: POST /bff/buildings
   - When building is updated: PUT /bff/buildings/{id}

3. Floor Cache
   - When floor is created: POST /bff/floors
   - When floor is updated: PUT /bff/floors/{id}

Implementation:
async def update_building(self, building_id, data):
    result = await self.adapter.update_building(building_id, data)
    await self.cache.delete_cached(f"building:{building_id}")
    return result

KEY POINTS
================================================================================

1. Cache is optional but recommended
   - Improves performance significantly (4-5x faster)
   - Reduces load on external services (80% fewer calls)

2. Cache invalidation is critical
   - Keep cache fresh by invalidating on updates
   - Use consistent cache key format

3. Use async Redis client
   - aioredis for non-blocking I/O
   - Consistent with FastAPI async design

4. Monitor cache performance

5. Test cache behavior
